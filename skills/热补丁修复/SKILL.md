# 热补丁修复

持久内核不能重启，但改代码不需要重启：**内核里开发，子进程里终审**。
活内核重绑定立即生效（试错），临时子进程从磁盘重读（复现"用户下次打开"）。两个进程各司其职，全程不重启对话。

**物理基础**：函数从不"背下"依赖——函数体里的 `os.walk`、`helper` 编译成 `LOAD_GLOBAL`，每次调用现去模块字典按名字取对象。所以**不改函数本体、只换它按名查到的那个东西，行为就变**。全部手法立足于此。

## 第一步：定位接缝，不是读懂全部逻辑

`inspect.getsource(目标)` 看源码，但**带一个问题读**：坏数据/坏行为是经由哪个"可替换的依赖"流过的？
- 例：grep 吐 history.json，因为 `os.walk` 给了它带噪声的 names → 接缝就是模块名 `os`。
- 认出接缝即可下手，不必读懂全函数。找不到接缝，读一千行也无从下手。

（要看项目整体结构/改动波及面，配合「项目图谱」skill。）

## 第二步：三种换法，按代价从轻到重逐级升级

1. **数据问题 → 改模块变量**（最轻）：`gm = sys.modules['tools.grep']; gm._EXCLUDE |= {...}`，调一次看行为变没变。变了→数据路线通，收。
2. **逻辑缺失 → 代理夹层**（核心手法）：换掉依赖对象，只拦一个出入口补缺的一步，`__getattr__` 把其余原样转发 → 零副作用。
   ```python
   class _CleanOS:
       def __init__(self, real): self._real = real
       def walk(self, *a, **k):                      # 只拦这一个
           for root, dirs, names in self._real.walk(*a, **k):
               yield root, dirs, [f for f in names if not _is_noise(f)]
       def __getattr__(self, n): return getattr(self._real, n)  # 其余全转发
   gm.os = _CleanOS(gm.os)
   ```
3. **散在中段、无法靠单一依赖隔离 → 包装函数本身**（最后手段）：旧函数调一次、改其输入/输出。

⚠️ 别依赖探索时留下的临时全局状态让修复"通过"——干净用户复现不了。

## 第三步：改对"层"，否则跨轮被冲掉

每轮启动会重注入预置函数。**改模块内部对象**（`tools.grep.os`）→ 注入够不着 → 存活；
改命名空间函数名（`grep`）→ 可能被盖掉。优先选能藏在模块内部的接缝——这也是代理夹层比包装函数更稳的原因。

## 第四步：落盘——两条路线，都要子进程对账

内核里那份未必等于磁盘那份，落盘后必须重新验。两条路线：

- **路线 A（代理夹层等行为修复）**：内核验证的是行为契约，落盘要**另写一份源码**——代理类直接落盘会变成多余的间接层（债），须翻译成源码语境最直接的写法（如直接在函数里加过滤）。内核版与落盘版是两份，全靠子进程终审对账。
- **路线 B（源码字符串）**：`getsource` 抓源码 → 字符串上精准改 → `exec(fixed, dict(fn.__globals__))` 出新函数验证 → **同一个字符串落盘**。内核跑的和落盘的同一份，零翻译误差。
  ⚠️ 字符串 `.replace` 失配会**静默不改**（比报错更危险）——改前必 `read` 核对真实字面。只对有真实源文件的函数成立。

落盘后 `read` 读回确认文本就是你要的。

⚠️ 路线 B 的 `getsource` 陷阱：如果内存里已经热补丁过，`getsource` 返回的是**当前内存版本**，不是磁盘原文。落盘时直接用 `read` 读磁盘确认字面，别走 `getsource`。

## 第五步：子进程终审（金标准）

全新进程从磁盘重读，零残留——必须在落盘后，因为它读的是磁盘：
```python
bash('python -c "import 模块; print(模块.功能(真实输入))"', cwd=项目目录)
```
这里对 = 用户下次打开就对。查 `.returncode` 看是否崩，读输出看是否正确。（多行脚本写临时 .py 再跑，别拼分号；`-c` 含 `<` `>` 或换行时 shell 转义必炸，直接用临时文件。）
子进程不带内核任何补丁——它才证明"磁盘源码本身对"，而非"内核内存恰好被打了补丁"。

## 第六步：回归

`bash('python tests.py', cwd=项目目录)` 确认没破坏别的。

## 坑：旧引用不自动更新

热补丁只对"调用时才查找"的引用生效：
- `模块.属性` 访问 → 感知更新 ✅
- `from 模块 import 函数`（名字已拷进别的命名空间）→ 仍指向旧的 ❌，需重新 import 或重绑
- 函数体内部才 import 的 → 每次调用重新查找，天生安全

拿不准时别信内核里的结果，信子进程——它根本没有旧引用。
