# 乐高式预制件架构 · 开发规范

> 快照ID: `LEGO_ARCHITECTURE_20260703`
> 生成时间: 2026-07-03
> 触发条件: 进行项目架构设计、代码重构、或规划AI辅助开发工作流时加载

## 依赖前提
已理解以下基础共识：
1. AI本质是"文本补全器"，不具备真正的长期记忆
2. 遵循"生成性即行"原则——AI在每次对话中生成新内容
3. 理解"记忆工程"的结构性悖论：试图让AI记住越多，AI越容易出错

---

## 不可逆核心决策
这些决策一旦接受，会永久改变思考方向，不再回头。

| 决策节点 | 结论 | 不可逆原因 |
| :--- | :--- | :--- |
| **架构范式的根本逆转** | 从"AI服务于项目架构"转向"项目架构服务于AI"。不再让AI去适应记忆系统的需求，而是让项目架构本身变得无需记忆。 | 一旦接受此框架，就不再从"如何让AI记住更多"的角度思考问题，而是从"如何让任务本身变得无需记忆"的角度设计工作流。 |
| **耦合度的终极目标** | 不是低耦合，是**零耦合**。每个模块是可移除的"预制件"，其缺席不应导致系统崩溃，最多只导致特定功能的缺失。 | 此目标决定了所有后续设计决策。接口设计、模块拆分、测试策略，全都围绕"可移除性"展开。 |
| **AI的角色定位** | AI是"预制件工厂"里的机器，负责在每次新对话中制造一块全新的、独立的积木。AI不负责记忆全局、不负责组装、不负责管理项目状态。 | 此定位决定了AI输入的设计原则：每次只注入当前模块的规格，不注入项目历史。 |

---

## 核心约束边界
1. **架构约束**: 采用六边形架构（端口与适配器）形态。核心业务逻辑是零耦合的"预制件"集合，所有外部技术（数据库、Web请求等）通过适配器接入。
2. **工程原则**: 严格遵守依赖倒置、接口隔离、单一职责、关注点分离、正交性等原则。
3. **模块通信**: 模块间只通过接口合同进行通信，不允许直接依赖内部实现。依赖方只知道接口，不知道具体实现。
4. **集成方式**: 遇到接口不兼容的预制件时，不修改预制件本身，而是编写"适配器"进行接口转换。

## 结构转折：核心观念转变

### 1. 从"全栈式开发"到"工厂式预制"
- 旧模式：在同一个上下文中，从一个庞大的、相互依赖的系统里"现场浇筑"功能
- 新模式：将每个功能需求，都视为一个独立的订单，在隔离的"工厂"（新对话）中完成"预制件"的加工

### 2. 从"记忆驱动"到"合同驱动"
- 旧模式：项目通过共享记忆（上下文）来协同，AI需要记住整个项目的细节
- 新模式：项目通过共享一组稳定的、精确的"接口合同"协同。合同就是预制件之间的唯一连接点，AI只需要理解当前模块的合同，不需要记忆全局

### 3. 从"理解系统"到"遵循合同"
- 旧模式：开发一个模块，需要理解整个系统的全景，认知负荷极高
- 新模式：开发一个"预制件"，只需要理解和遵循它的接口合同，不需要理解其他模块的实现

## 核心理念澄清

### 什么是"零耦合"？是不是完全没有依赖？
**零耦合不是没有依赖，而是依赖抽象，不依赖具体实现**：
- ✅ 模块可以依赖接口合同
- ❌ 模块不能依赖具体实现的内部细节
- 结果：修改具体实现，只要接口合同不变，所有依赖方都不受影响，不会产生连锁反应

### 接口到底是什么？为谁设计？
- 接口不是复杂的框架概念，它就是一张**服务合同**
- 合同只说清楚三件事：你给我什么参数，我给你什么结果，我能给你提供什么服务
- 不说清楚我具体怎么实现这个服务
- 接口为**双方**设计，但核心是保护依赖方：
  - 对依赖方（使用者）：只需要看合同，不需要看实现，改实现不影响你
  - 对被依赖方（提供者）：只需要遵守合同，不需要关心谁用你，你也不需要关心使用者的内部

### 什么时候需要定义接口？
| 场景 | 是否需要接口 | 原因 |
| :--- | :---: | :--- |
| 被多个模块依赖 + 容易变化 | ✅ 必须定义 | 改一处不用改多处，收益极大 |
| 只被一个模块依赖，但未来可能换实现 | ✅ 值得定义 | 成本低，未来替换方便 |
| 只被一个模块依赖，且基本不会变 | ❌ 不需要 | 过度设计，收益抵不上成本 |
| 已经是稳定的统一载体（如`result`） | ✅ 已经是接口，不需要重新定义 | 它本身就是合同，所有依赖方只使用稳定的公共属性 |

### 接口能解决AI辅助开发的什么问题？
- 解决AI"记不住全局面"的本质问题：AI每次只需要理解当前模块和它依赖的接口合同，不需要记住整个项目
- 避免连锁错误：改一个模块不会意外影响其他模块
- 符合"生成性即行"：每次对话只生成一块积木，拼起来就是完整系统

## 核心操作流程

### § EXTRACT · 抽离（拆除模块）
面对复杂类或函数，不整体重写，逐块拆除：
1. 识别一个内聚的逻辑块，以它的"意图"命名，创建一个新的模块
2. 将逻辑块安全移入新模块，只通过参数和返回值与原位置交互
3. 原位置替换为对新模块的调用
4. 立即测试验证，确保系统行为不变
5. 重复以上过程，每次只拆一块，持续将大模块分解为多个零耦合预制件
6. 拆除后检查：接口是否简单，内部是否高内聚，是否消除了重复概念

### § INTERFACE · 接口（设计预制件接口）
为预制件设计接口：
1. 定义完整合同：必须包含三要素——语法（方法签名）、语义（状态变化）、协议（调用顺序）
2. 最小化暴露：接口只暴露客户端需要的功能，隐藏所有内部实现细节
3. 保持抽象一致：检查接口中的方法，确保它们都处于同一个抽象层级，将异类方法拆出
4. 单一关注点：一个接口只服务于一个核心关注点，多个关注点则拆分为多个独立接口
5. 依赖抽象，不依赖具体实现：模块依赖接口，不依赖具体实现
6. 遇不兼容预制件时，不修改预制件，编写适配器转换接口

### § SPECIFY · 标定（需求规格化）
需求对齐后，将共识转化为书面规格：
1. 每条功能用"系统必须…"句式，包含前置条件、后置条件、可测量的验收标准
2. 不写"快速""便捷""用户友好"等模糊描述，写具体数值和判定方法
3. 撰写执行摘要：问题是什么，方案是什么，需要什么资源，下一步做什么决定
4. 逐项检查：是否完整、正确、无歧义、可验证、一致、可追溯
5. 任何一条未通过则退回重写，不部分通过
6. 规格确认后锁定，成为后续执行和验收的唯一基准，修改须重新走标定流程

### § INFUSE · 注入知识（开发前准备）
接到编程任务后，不直接写代码：
1. 先解析核心意图与潜在变化点
2. 从设计模式库中匹配最相关的模式（适配器/策略/工厂等）来封装变化
3. 注入架构原则（分层解耦·DRY·高内聚）与命名规范（名副其实·单概念一词）作为生成约束
4. 生成代码后逐项检查：是否高内聚、低耦合、可扩展、无冗余、命名精确无歧义
5. 不通过则应用重构手法（提炼函数·消除重复）进行小步改造，每一步可回滚
6. 整个过程中确保测试保护

### § ELEMENT · 元素（拆解系统）
将复杂系统拆解为彼此独立之最小单元：
1. 每单元隐藏一个设计决策，该决策若变只变此单元
2. 单元接口稳定，不暴露内部实现，内部可变而外部不感
3. 高内聚：单元内全部元素服务于同一功能，不杂
4. 低耦合：改一单元不涉他单元，内部状态不被外部依赖
5. 单元各自完整，自有触发条件与退出条件
6. 可通过接口自由组合，顺序可重排，组合可变化

### § DECOMPOSE · 解构（分解难题）
把难题分解为尽可能多的简单部分，从最简单者开始，逐步上升：
1. 连续步骤彼此衔接，每一步简单到可可靠执行，每一步服务于整体
2. 找到内部相互作用较弱的边界，在此处下刀
3. 分解至每一部分微不足道，组合时暴露先前分解之误，循环重分

### § SEQUENCE · 序列（编排步骤）
将整体分解为严格先后之步骤，次序不可逆：
1. 每一步只做一个操作，完成后方可进入下一步
2. 上一步的输出即下一步的输入，不跳步，不并行
3. 检测跳步：若某步依赖之前未完成之输出则阻断
4. 先完成上游再进入下游，上游未完成而下游先做则返工
5. 偏差进入的那一步即需要改进的那一步，不改进全体

## 验收标准（乐高积木测试）
每个预制件必须通过这三个测试，才算合格：

### 1. 可移除性测试
- **测试方法**：将这个模块的代码从项目目录中移走
- **验收标准**：项目应能继续启动，且不因该模块的缺失而崩溃。最多只缺失该模块提供的功能。

### 2. 独立性测试
- **测试方法**：为这个模块编写单元测试
- **验收标准**：测试的`setUp`环节不应需要初始化任何其他模块的实例，整个测试环境完全由该模块本身和必要的Mock对象组成。

### 3. 合同一致性测试
- **测试方法**：尝试替换实现
- **验收标准**：任意一个满足了接口合同的实现，都应该能够无缝替换现有模块，且系统行为不变。

## 收益总结
1. **对AI友好**：每次只需要理解一个模块和它的接口合同，不需要记忆全局，AI出错概率大幅降低
2. **可渐进重构**：不用一次性推倒重来，每次拆一块，系统始终可运行
3. **零连锁反应**：改一个模块不会影响其他模块，维护成本大幅降低
4. **可替换可扩展**：新实现只要符合接口，就能无缝替换原有实现
5. **可独立测试**：每个模块都能独立测试，质量更有保障

## 当前项目优先级（示例）
基于本规范，当前项目的重构优先级：
1. `call`模块：5次依赖，易变，优先级最高，先定义`LLMCaller`接口
2. `compact`模块：依赖`call`，是完整独立功能，优先级第二，定义`HistoryCompressor`接口
3. `history`模块：4次依赖，稳定性高，后续处理
4. `result`模块：15次依赖，但本身已是稳定合同，不需要重构

## 目录结构规范

调整后的项目目录按"接口-实现-核心-公共"分层，结构清晰，职责明确：

```
<project_root>/
├── ports/                   # 所有接口（端口），只放接口定义，不放具体实现
│   ├── llm_caller.py        # LLM调用接口
│   ├── history_compressor.py # 上下文压缩接口
│   ├── chat_history.py      # 对话历史接口
│   └── ... 其他所有接口定义
├── implementations/         # 具体预制件实现（含适配器，小项目不需要额外分层）
│   ├── default_llm.py       # 原有call模块的实现，直接实现LLMCaller接口
│   ├── default_compressor.py # 原有compact模块的实现，直接实现HistoryCompressor接口
│   ├── openai_llm.py
│   └── ... 其他具体实现
├── core/                    # 核心入口和组装代码
│   ├── agent.py
│   ├── run.py
│   ├── inject.py
│   ├── load.py
│   └── ... 核心流程代码
├── common/                  # 公共基础组件（稳定不变的统一契约）
│   ├── result.py            # 统一结果载体，本身已是契约
│   ├── manifest.py
│   └── ... 其他公共基础
├── tools/                   # 预置工具模块
│   ├── bash.py
│   ├── edit.py
│   └── ... 其他工具
└── skills/                  # 技能模块（动态扫描）
```

### 目录职责说明
| 目录 | 职责 | 放置内容 |
| :--- | :--- | :--- |
| `ports/` | 接口定义 | 只放抽象接口（ABC），只有方法签名，没有业务逻辑实现 |
| `implementations/` | 具体实现 | 所有预制件的具体业务逻辑实现，直接实现对应接口。如果需要适配已有实现，直接在这个目录里放适配器就行，小项目不需要额外分层 |
| `core/` | 核心流程 | 系统启动、组装、主流程编排代码 |
| `common/` | 公共基础 | 稳定的基础组件，被多方依赖，本身已是统一契约，不需要再抽接口 |
| `tools/` | 预置工具 | 暴露给AI使用的工具函数 |
| `skills/` | 技能模块 | 动态扫描的技能，保持原有结构 |

### 优点
- 接口和实现彻底分离，找接口直接去`ports/`，一目了然
- 完全符合六边形架构（端口与适配器）的设计思想
- 结构清晰，人也好找，AI也好找
- 适合当前项目规模，不会过度设计，也不会太粗糙
