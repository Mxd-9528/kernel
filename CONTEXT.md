# CONTEXT

IPython 是持久内核——提供无限循环、持久命名空间、异常捕获、表达式值捕获。
系统由单一 Python 进程构成。所有逻辑均为 IPython 命名空间内的普通 Python 函数，不存在独立于内核的架构层或特权对象。

**元决策：同进程单例（不是子进程内核）。** 这一个选择支配了下游一连串结局，后人不必反复重新发现：
- 活对象在手 → 注入用 `shell.push`、看实现用 `inspect`，都是直接摸对象（优势）。
- 模块缓存常驻 → 改磁盘代码要让其生效，须退出重开真新进程，`restart` 式重置做不到（代价，见约束 8）。
- 难做强隔离 → 子代理这类"干净环境"需求只能靠子进程（`bash('python -c ...')`），不能靠重置内核。

对比废弃的旧项目（jupyter 子进程内核）：它的臃肿一半来自子进程带来的注入/通信管道。同进程换来极简，代价是上面这些——值得，因为绝大多数功能不需要强隔离。

## 约束

1. **无架构层**：所有组件（编排、工具、入口模块）均为 IPython 命名空间中的一等 Python 对象，通过标准 `import` 或命名空间查找通信。`chat.py` 是入口模块，无特权。
2. **依赖获取**：函数通过 Python 标准库或 IPython 命名空间运行时查找获取依赖。禁止硬编码同级模块文件路径或假设其内部实现。
3. **唯一特权是「导入持久内核」**：真正的地基是持久内核，内核本身不可触碰。除了「被注入内核命名空间」这个动作，没有任何模块有特权——编排、工具、加载，全是普通 Python，怎么组织都自由。三个同构的约定驱动发现层（零配置代码）：`load.py`（机件：加一行 import → dir 扫描自动注入内核）、`manifest.py`（工具：在 `tools/` 放一个带同名函数的文件 → 自动注入内核 + 自动生成 prompt 文档）、`skills.py`（技能：建一个带 SKILL.md 的目录 → 自动索引进 prompt）。统一架构哲学：最小的接口获得最大的杠杆。
4. **执行路径唯一**：模型输出的代码块与预置函数通过同一条 `run()` 路径执行，返回同一套三元组反馈。
5. **反馈接口统一**：所有执行反馈通过 `Result` / `ListResult` / `DictResult` 返回。Body 保持原生 Python 类型行为。判断成败查 `.error`，不要用 `if not r:`。
6. **不重造 IPython**：IPython 已提供的持久化、循环、异常处理、magic 等，不做二次实现。
7. **最小预置**：只预置模型无法用单次简单调用完成的功能。标准库一行可完成的，不封装。
8. **模块可触碰**：所有模块可通过 `inspect` 标准库获取源码、签名、文件路径。模型改磁盘文件后，需退出重开 `python chat.py`（真新进程）才让新代码生效——同进程内 `sys.modules` 缓存会让重新 import 拿到旧对象。要临时清内核变量，模型自己写 `%reset -f` 或 `globals().clear()` 即可，不预置专门函数。
9. **兼容 Python 3.8**：不用 3.9+ 语法（如 dict `|` 合并、`list[int]` 内置泛型注解、`removeprefix`）。ruff 按 `py38` 检查会自动揪出。兼容低版本只扩大可运行环境，无成本。

## 开发方法论

真正的架构不是那些 .py 文件，是产出它们的两条元规则：

**判断漏斗——每个"加功能/改代码"的决策都过这三道闸：**
1. 这需要存在吗？（YAGNI）不需要就砍，并说明理由。
2. 已有的能做吗？标准库 / 已装依赖 / 内核里的活对象，能用就用，别造。
3. 旧项目怎么做的？萃取其精华，弃其包装——它的臃肿多来自子进程管道和缺少这道漏斗。

**踩坑即前置——这是本项目最有效的引擎：** 模型每次踩的坑（如：不知道自动回显、误把 returncode 当工具失败、把 read 的行号抄进 edit、`getsourcelines[1]` 丢源码），都应把教训写进它**下次开局就会看到的地方**（`prompt.txt` 或对应工具的 docstring）。让下一个模型开局即有这份认知，而不是再撞一次。

**实测 > 推断：** 凭印象下的结论反复被推翻过（reload、`__repr__`、bash 行为）。涉及运行时行为的判断，跑一次真内核/子进程验证，别凭推理拍板。
