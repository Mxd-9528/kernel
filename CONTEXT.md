# CONTEXT

## 元决策：架构服务于 AI，不是反过来

AI 无状态、每次从零开始。主流路线是**加大记忆**（更长上下文、知识图谱、多层摘要）——但记忆必然压缩有损，而真正致命的地基锚点是**隐含的、事前不可穷举**，一旦丢失就永久跑偏。

结论：**不改造 AI 去适应大项目，改造项目让任务本身不需要记忆。**

## 手段：沉淀线

项目分两层可读性：

- **水线以上**：每次对话都读（编排、工具、技能、合同）
- **水线以下**：除非改它本身，否则不读（合同已经描述了它的对外行为）

**合同 = 沉淀线**。对被依赖模块，在旁边加一份 `<模块>_contract.py`：10-25 行讲清对外签名、返回、失败模式、跨层协议。上游 `from <模块>_contract import ...`，实现自由变，合同不变即零波及。

**准入门槛（缺一不做）**：
1. 功能收敛，能力清单基本不再变
2. 合同可完整表述（10-25 行覆盖对外全部行为）
3. 上游只关心合同（不需要继承、不需要摸内部）

已沉：`call_contract.py`、`background_contract.py`。`result` 因上游继承类本体，物理不可分，**不沉**。`compact/history/agent/extract` 消费者太少或是编排层，**不沉**。

## 同构发现（第四层）

三层已有：`load.py`（机件）、`manifest.py`（工具）、`skills.py`（技能）。
第四层：`*_contract.py` 兄弟文件模式——最小接口（加一份合同 + 改 import），最大杠杆（上游读 10 行代替读 200 行）。

命名空间里 `[k for k in globals() if k.endswith('_contract')]` 就是"已沉淀模块"清单，可枚举、可 introspect。

## 元决策：同进程单例（非子进程内核）

IPython 提供持久命名空间、循环、异常捕获、表达式值捕获。所有逻辑均为命名空间内的普通 Python 函数，无特权层。

代价：改磁盘代码须退出重开真新进程；强隔离靠子进程（`bash('python -c ...')`），不靠重置内核。收益：注入用 `shell.push`、内省用 `inspect`，直接摸活对象。

## 约束

1. **无架构层**：编排、工具、入口全是命名空间一等对象，`chat.py` 无特权。
2. **执行路径唯一**：模型代码块与预置函数走同一条 `run()` 路径，同一套三元组反馈。
3. **反馈接口统一**：`Result` / `ListResult` / `DictResult`，判断成败查 `.error`，不用 `if not r:`。
4. **不重造 IPython**：持久化、循环、异常处理、magic 已有的不做二次实现。
5. **最小预置**：标准库一行能做的不封装。
6. **模块可触碰**：所有模块可 `inspect` 拿源码/签名/路径。改代码须重开进程。
7. **Python 3.8**：不用 3.9+ 语法（`dict |`、`list[int]`、`removeprefix`）。ruff `py38` 自动检查。

## 开发方法论

- **判断漏斗**（每次加/改代码都过）：① 需要存在吗？(YAGNI) ② 已有的能做吗？(标准库/依赖/命名空间活对象) ③ 能沉到水线以下吗？(合同覆盖能否让上游不下潜)
- **踩坑即前置**：模型踩过的坑写进 `prompt.txt` 或工具 docstring，让下一个模型开局即知。
- **实测 > 推断**：涉及运行时行为的判断，跑一次真内核验证。
