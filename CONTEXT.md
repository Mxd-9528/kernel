# CONTEXT

本项目的设计思想沿用八本经典软件工程文献。本文档说明这些原则在"调用者为无状态大语言模型 (LLM)"这一场景下如何具体化落地。

## 参考文献

- Parnas, D. L. (1972). *On the Criteria To Be Used in Decomposing Systems into Modules.* —— 信息隐藏 (Information Hiding)。
- Gamma, E., Helm, R., Johnson, R., Vlissides, J. (1994). *Design Patterns.* —— *Program to an interface, not an implementation.*
- Martin, R. C. (1996 / 2000). *SOLID Principles.* —— 依赖倒置原则 (Dependency Inversion Principle, DIP)。
- Fowler, M. (1999 / 2018). *Refactoring: Improving the Design of Existing Code.* —— 主动消除多余的解释，使代码本身自明。
- Beck, K. (1999). *Extreme Programming Explained.* —— YAGNI (You Aren't Gonna Need It)。
- McConnell, S. (2004). *Code Complete, 2nd Edition.* —— 简洁性作为设计目标；拒绝"以备不时之需"。
- Maeda, J. (2006). *The Laws of Simplicity.* —— 删除到失去意义那一刻停下，即是完成。
- Ousterhout, J. (2018). *A Philosophy of Software Design.* —— 深模块 (Deep Module)、认知负荷 (Cognitive Load) 最小化。

## 元决策：少读一点

**表层约束**（无状态 LLM 是新场景）：传统软件工程假设读者是人类工程师，具有跨会话记忆。本项目的读者是 LLM——每次会话从零开始，不保留上一次的上下文。由此产生一条硬性约束：**每次会话所需读取的代码量应当最小化**。方法不是扩大 LLM 的记忆容量（更长上下文、检索增强、多层摘要），而是让代码本身不需要那么多记忆。

**深层原则**（沿用经典）：减少认知负荷是设计目标本身，不是各具体规则的副产品。这一原则在多位经典作者处以不同视角出现：

- **Parnas 1972**："一个被隐藏的决策，免除了每一个不直接负责该决策的开发者去理解它的义务。" 隐藏 = 阅读权利的行使，不只是依赖切断。
- **Fowler 1999**："当你感觉需要写一条注释时，先尝试重构代码，让注释变得不必要。" 目标不是"读得少"，是"不需要读更多"——从被动忍受升级为主动追求。
- **McConnell 2004**："保留'以备不时之需'的东西——不时之需永远不会按预期到来，而它在等待中持续消耗认知资源。"
- **Maeda 2006**："删除到你觉得它开始失去意义的那一刻，然后停下来。你停下的那个点，才是真正完成的时刻。" 完成的判据是删无可删，不是加无可加。
- **Ousterhout 2018**：认知负荷是模块深度、接口设计、注释密度的统一度量——任何增加读者负担而不提供等价价值的选择，都是设计的失败。

传统软工中这几条已是最佳实践；LLM 无状态场景把"最佳实践"变成"硬性约束"——**过度阅读不只是低效，是不可能**。

以下所有章节都是这一元决策的具体化——信息隐藏、深模块、静态判据、YAGNI、不封装清单，都在服务同一件事：让读者读得少，且读到的每一样都自足。

## 1. 信息隐藏 (Information Hiding, Parnas 1972)

**原理**：一个模块 = 一个被隐藏的设计决策 (Parnas)。模块的价值取决于它藏起了什么——决策越复杂、越易变化，抽象价值越高。

Parnas 原文：

> "We propose that one begins with a list of difficult design decisions or design decisions which are likely to change. Each module is then designed to hide such a decision from the others."

GoF 的 "Program to an interface, not an implementation" 与 Martin 的 DIP 都是同一原理的不同视角——调用方依赖抽象，实现服从抽象。三者在本项目里合为一条规则：**上游只写 `from call import call`，禁止 `from _call import call`**。接口与实现冲突时改实现对齐接口，除非架构师明确修订接口。

**判断一个模块是否值得封装为独立接口**（准则，非规则）：

- **它藏起了什么设计决策？** 若没有值得藏的决策（平铺编排、标准库薄封装），封装无收益。
- **这些决策是否稳定？** 决策仍在剧烈演化时，接口追着实现改，隐藏失效。
- **调用者是否需要绕过接口去理解决策？** 若必须继承实现类、访问私有属性、感知内部数据结构，抽象未真正生效，是浅模块。
- **有多少调用者依赖这些决策？** 多调用者时隐藏防止决策扩散；单调用者时退化为该文件内部组织。

**落地**：项目分两层文件：

- **接口层**：`call.py` / `background.py` / `compact.py` / `run.py`。文件名无下划线前缀。内容为 10–25 行的公开 API 声明（docstring 描述签名、返回值、失败模式、跨层协议），末尾通过 `from _xxx import ...` 从实现层重导出符号。
- **实现层**：`_call.py` / `_background.py` / `_compact.py` / `_run.py`。文件名带下划线前缀 (PEP 8 私有命名约定，同 CPython 的 `_ssl` / `_socket` / `_json`)。承载全部实现细节。

## 2. 深模块 (Deep Module, Ousterhout 2018)

**原理**：模块的抽象价值 ≈ 隐藏的实现复杂度 / 暴露的接口复杂度。接口窄、实现厚的模块 (深模块) 抽象价值高；接口面积接近实现面积的模块 (浅模块) 未减少调用者认知负荷，反而增加一层间接性。

深度是信息隐藏的量化视角——衡量的正是"藏起了多少东西"。

**本项目已封装的四个深模块**：

| 模块 | 隐藏的设计决策 |
|---|---|
| `call` | HTTP 请求组装、鉴权、.env 加载、reasoning_content 剥离 |
| `background` | 线程池调度、超时监控、结果持久化、任务状态机 |
| `compact` | 对话历史拆分、轮次计数、结构化压缩提示词、事实/提炼两层方法论 |
| `run` | IPython InteractiveShell 生命周期、stdout/stderr 捕获、ANSI 清洗、Result 分类 |

## 3. 静态可验的依赖判据

**原理**：面向接口编程的合规性应当可以静态检查，不依赖运行时行为观察。

**落地**：`tests.py` 用正则扫描所有 `.py` 文件；命中 `from _xxx import` 或 `import _xxx` 即为封装泄漏 (encapsulation leak)。接口文件本身与 `tests.py` 豁免（前者是唯一合法转手点，后者拥有跨层测试特权）。签名一致性由对象同一性断言 (`call.call is _call.call`) 检测。

## 硬约束

1. **执行路径唯一**：模型生成代码块与预置函数走同一条 `run()`，返回同一套 Result 三元组。同进程单例，改磁盘代码须重启进程；强隔离通过子进程实现。
2. **反馈接口统一**：成败查 `.error is None`，不用 `if not r:`（`Result("")` 为假但代表成功）。
3. **约定优于配置**：`tools/*.py`、`skills/*/SKILL.md`、接口层文件命名 —— 通过文件系统扫描/命名约定自动接入，新增预置函数或技能仅需创建文件。
4. **模块可反射**：所有模块与函数可通过 `inspect` 获取源码、签名、路径。
5. **不重造 IPython 已有能力**：持久化命名空间、REPL 循环、异常捕获、magic 命令。
6. **最小预置**：Python 标准库一行代码可完成的操作不封装为预置函数。
7. **Python 3.8 兼容**：不使用 3.9+ 语法；`ruff` 配置 `target-version = "py38"` 自动检查。

## YAGNI (Beck 1999)

每次增删代码前的思考顺序：

1. 此改动当前是否有明确调用需求？没有则不写。
2. 若需要，标准库、已引入依赖、命名空间中已存在对象是否可以直接完成？可以则不新增代码。
3. 若必须新增，回到 §1 的判断决定平铺还是封装。

## 反模式：不采纳的深化机会

判断改动是否值得做，除正面判据（§1 信息隐藏、§2 深模块）外还有反面模式——**看似深化，实为增负**。以下模式来自经典软件工程文献与本项目实践，用于快速识别错配的抽象诱惑：

| 反模式 | 触发场景 |
|---|---|
| **过度抽象 (Over-abstraction)** | 为已可局部理解的功能引入间接层——接口面积接近实现面积 (Ousterhout 浅模块)、编排步骤拆至新模块而复杂度不减、或将不同领域的相似结构合并 (Metz "Wrong Abstraction")。 |
| **推测通用性 (Speculative Generality, Fowler 1999)** | 为可能未来发生、当前不存在的需求引入参数、配置或扩展点。判据：当前无第二调用者、无实际变更压力、无已发生的痛点。常见伪装形态包括"可测性优化"、"并发安全预留"、"可扩展性铺路"——判据不因动机的正面性而放松。 |
| **过早统一 (Premature Unification)** | 为消除表面差异（重复行、透传函数、目录形态不齐）而破坏既有约定或引入特殊规则。收益是审美齐整，代价是约定复杂化。 |
| **误导性命名 (Misleading Names)** | 用领域内已有严格语义的重词（如 prompt、agent、context）命名字面上更简单的功能。读者被术语引至更复杂的理解，实际功能仅是其字面意义。 |
| **拟主体谬误 (Agentic Fallacy)** | 将用户主动动作包装为 LLM 的"自动调度"或"按需触发"能力。LLM 无内部状态、无跨调用记忆，所谓"按需"永远是用户手动触发的委婉表述。判据：功能依赖模型识别处境并自主选择行为，且处境无法由单一关键词匹配。 |

反模式非绝对禁令。当原理判据压过反模式警告时改动仍可进行——反模式是**默认警戒**，不是**否决权**。

命中反模式的提议须显式对齐：列明命中项名与压过判据；未显式对齐视为未识别命中。这一要求抵消评审工具的天然偏见（发现深化机会 = 系统性推荐过度抽象）。

## 不封装的模块

| 模块 | 原因 |
|---|---|
| `result.py` | 调用者继承类本体；类的公开面即使用方式，无独立可隐藏的决策 |
| `history.py` / `agent.py` | 编排层或薄封装；内部无复杂易变决策 |
| `tools/*.py` | 通过 IPython 命名空间注入调用；`inspect.signature` + docstring 已完成信息隐藏 |

## 预置函数与模型代码块的能力等价性

预置函数（`tools/*.py`）与模型现场书写的 Python 代码块能力等价：预置函数本质是"命名的代码块 + 深思熟虑过的实现"，不引入新能力。其价值仅在于减少每次调用的重复书写（少写）、减少调用者理解成本（少读）、复用已固化的边界条件处理（如 `read` 的 offset/limit、`edit` 的唯一性检查）。任何未预置的能力，模型可通过在代码块内直接调用 Python 标准库或 subprocess 达成，等价可及。此原则用于抵制"为增加能力而新增预置函数"的诱惑——增加预置函数的判据是使用频率与深思熟虑成本，不是能力覆盖。
