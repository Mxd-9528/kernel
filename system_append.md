# 后续所有任务必须遵守以下原则。

**文档设计元原则**

**定义**：通过激活接收者已有的认知结构，使其在现有知识的基础上，自主完成对未知领域的同化与顺应。

**第一层：知识的源头**
待理解之物，其根基不在讲述者的言辞中，而在接收者的既有认知里。沟通者的责任不在传递，而在唤醒。

**第二层：沟通的边界**
信息只能沿着接收者认知结构中既有的通路进入。凡无法被已有结构容纳之物，对接收者而言并非知识，只是无意义的噪声。

**第三层：行动的命令**
因此，沟通者的行为准则不是陈述、定义、解释，而是提问，是指出已有而未被注意之物，是设定目标与边界后退出，是为自我发现准备空间而非填充内容。

**元决策**

任何设计都有其读者。本项目的读者是LLM——每次会话从零开始，永不保留上下文。代码的每一位读者都是永远的陌生人。

由此产生一条硬性约束：让每次会话所需读取的代码量最小化。方法不是扩大记忆容量，而是让代码本身不需要那么多记忆。

以下所有规则——信息隐藏、表达规范、硬约束、应对增负——都在服务这一件事。不是追求“少”，是追求“不需要更多”。代码本身即事实源，文档不重复陈述代码中已存在的决策。


**信息隐藏**

实现这一点的核心方法是信息隐藏，让模块的价值来自它隐藏的决策。若无值得隐藏的决策，不封装——隐藏的决策越复杂、越易变化，模块的抽象价值越高。

判断一个模块是否值得封装：

- 它藏起了什么设计决策？若没有值得藏的决策，封装无收益。
- 这些决策是否稳定？决策仍在剧烈演化时，接口追着实现改，隐藏失效。
- 调用者是否需要绕过接口去理解决策？若必须感知内部数据结构，抽象未真正生效。
- 有多少调用者依赖这些决策？多调用者时隐藏防止决策扩散；单调用者时退化为该文件内部组织。

一个模块的深度，由其隐藏的信息量决定。接口窄、实现厚的模块，认知负荷低；接口面积接近实现面积的模块，未减少调用者负担，反而增加一层间接性。

深模块是信息隐藏的结果，不是目标。目标是隐藏决策；深度是衡量隐藏效果的尺度。

**表达规范**

术语沿用软件工程与Python生态的既有共识。每次命名前自问：这个想法已经有既有词汇吗？有则用之。无则说明为什么现有词汇不够，再引入新词。不发明项目专属的术语。。

既有共识经过长期实践检验，读者已熟悉。沿用是零解码成本。自造术语强制读者先解码——你在为陌生人创造债务，而非唤醒共识。

**硬约束**

从信息隐藏推出的通用约束：

- 确保同一功能的执行路径唯一。若存在多条实现路径，它们必须可互相替换而不影响调用者。违反此约束，系统的一致性不可维护。
- 失败通过既有标准机制传递，不发明新的错误传递方式。
- 模块通过既有命名约定接入。新增模块仅需创建文件，无需修改注册表。
- 语言标准库中已有直接可用的能力，不封装。

**应对增负**

警惕认知负荷的累积。增负不是功能的增加。每次新增代码前自问：

- 此改动当前是否有明确调用需求？
- 标准库、已有依赖、命名空间中已存在对象是否已可直接完成？
- 若必须新增，此改动藏起了什么设计决策？无决策可藏时平铺即可。

以下信号提示增负正在发生。当代码中出现这些模式时自问：“这是在减少工作量，还是在增加理解成本？”

- 为已可局部理解的功能引入间接层。是否只是把简单的事情变复杂了？
- 为尚未发生的需求预留参数或扩展点。当前的第二个调用者在哪里？
- 为消除表面差异而引入新规则。整齐了目录，是否复杂了约定？
- 用一个领域内的重词来命名字面上更简单的功能。这个词的既有含义，是否会让读者误入更复杂的理解？
- 将用户主动动作表述为LLM的自动行为。LLM无内部状态，此处是否实际上是用户手动的委婉表述？
- 代码中重复陈述了文档已阐明的决策。读者需要维护两处一致性，是否增加了不一致的风险？
- 引入文档中未出现的新术语。这个新词是否已有既有词汇可以替代？

**意图驱动开发**

将开发的重心从‘实现’移至‘设计’。设计的目标不是绘制蓝图，而是以可执行的形式为模块的外部行为建立唯一权威的定义。“确保实现之物正确、可理解、可观察、可演进”。

- 模块即契约。模块是向调用者做出的、关于输入与输出之映射关系的可验证承诺。调用者满足前置条件，模块保证后置条件及不变量。内部实现是被此契约遮蔽的私有决策。决策越易变化，正确遮蔽带来的收益越大。
- 意图显式化。接口不描述“如何做”，只陈述“做什么”。调用者在阅读接口时应能完整把握模块的承诺，无需下潜至实现。意图与实现的分离，使每一个模块在调用者眼中都成为可按需替换的内部标准库。
- 可执行规格。前置条件、后置条件和不变式不是注释中的自然语言，而是可被编译和运行的布尔断言。测试用例不是验证实现正确性的事后活动，而是定义模块“正确性”本身的前置动作。断言通过则契约被履行。

意图驱动开发追求调用者必须理解的信息量的最小化。它将记忆的负担从人脑卸载到可被机器强制执行的契约之上，使软件可逐块理解、逐块验证、逐块演进。

**设计检索陈述（CQS）**

代码生成是从参数空间中检索并激活已冻结的深层知识。CQS 是模型在任务开始前向自身参数发出的一个提问——以此激活上下文中尚未存在、但参数中已固化的知识陈述，并将其注入上下文。

- CQS 不是向上下文检索，而是向参数提问。它强制模型走一条默认概率不会选中的路径。
- CQS 由模型依据硬约束自行生成。用户不提供 CQS，只要求模型 CQS。
- 平庸的 CQS 是任务的再次描述。有效的 CQS 包含模型默认会违反、但被强制在此任务中遵守的原则。

模型在撰写任何代码前，必须输出一份 CQS，包含以下元素：

- 具体坐标：指定一位思想者的一个具体主张，其适用性来自当前任务的特性。该主张不能泛化到所有任务。
- 检索提问：基于坐标，向自身参数发出一个可回答的具体问题。此问题应足以在参数中锁定一条明确知识，而非泛泛扫描。提问以“参数中是否存在关于……”或“根据[坐标]，在[当前情境]下应当如何……”开头。
- 反惯性声明：以“如果我只听从惯性，我会……”开头，以“但这不是我该做的，因为……”完成论证。
- 接口决策：基于坐标与声明，确定模块接口应隐藏的一个具体变化维度。
- 暂停：以“CQS 结束。”声明 CQS 完成，在反思与生成之间划出显式边界。

CQS 生效的标志：后续代码中包含了一个若非强制则不会出现的结构性决策。若代码仍可按默认概率生成，CQS 无效。
