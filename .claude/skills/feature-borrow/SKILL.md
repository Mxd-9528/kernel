# feature-borrow —— 外部功能借鉴工作流

## 核心原则

**提炼洞察，不移植代码**。外部项目的功能实现是其整体架构的伴生品——重型工具的沙箱、多前端、并发、权限系统等大量复杂度不属于本项目。借鉴的正确姿态是提取"问题-方案对"（Christopher Alexander），用本项目的手法重新实现。

避免两个极端：**闭门造车**会引入未经验证的坑（模型的想象常与成熟设计有偏差）；**完全照搬**会带走不属于 kernel 的架构假设。中间路径是**教科书式提炼**——保留原理，删除架构伴生品，用 kernel 简洁语气重写。

## 工作流（六步）

### 1. 明确目标

只借鉴一个具体功能。范围过宽会导致 Explore 报告噪音过大。用户请求形态：
- "从 Codex 学 X"
- "研究 Cursor 的 Y 是怎么做的"
- "看看 Aider 的 Z 值得借鉴什么"

若用户请求偏泛（"学 Codex 的优点"），先与用户对齐到一个具体功能。

### 2. 派 Explore agent 深度探索

用 Agent tool 的 `Explore` 子代理探索外部项目。子代理任务清单：

- 定位相关代码路径（Grep 关键词 + Glob）
- 提取工具/接口的**完整对外描述**（原文照抄，含 docstring 与 prompt 中的规范说明）
- 描述数据结构、状态机、错误处理、特殊场景
- 找出源码注释、commit 消息、TODO/FIXME 中揭示的**踩过的坑**
- 列出关键文件路径 + 每个文件一句话职责

子代理返回 1000-2000 字详细报告。**不接受"简短摘要"**——本步骤的价值就在于详实。让子代理承担阅读量，主对话只保留结论。

### 3. 用反模式表 + "少读一点" 过滤

对子代理报告里的每个设计点，逐条判断"是否值得引入 kernel"。判据：

- 命中 CONTEXT.md §反模式 任一条 → 默认否决
- 已被 kernel 现有机制覆盖 → 不重复实现
- 属于重型工具架构伴生品（多前端、沙箱、并发、权限系统） → 不引入
- 属于对具体痛点的成熟经验 → 保留候选

生成一张"保留 / 合并 / 删除"的判决表——每个设计点都有明确归属。

### 4. 提炼三原则

对"保留候选"应用以下三条，压缩到教科书密度：

- **合并同源判据**：外部项目的规范条目常是同一原理的多角度表述。举例：Codex 的 update_plan 有 7 条"何时用"，其中 3 条本质是"任务非平凡"的具体化，合并为 1 条。
- **删除废话与元评论**：如"用户要 TODO 就用"（废话）、"理解变化时立即更新"（太抽象无操作性）。
- **示例只留对照**：外部项目常保留多个示例用于训练，个人工具留 1 高质量 + 1 低质量对照即可——对照的价值在语义对比，不在数量。

### 5. 生成候选实现

用 kernel 现有惯例落地：

- 若是预置函数，放 `tools/*.py`，走 manifest 自动发现
- docstring 是**规范单一事实源**（manifest 会自动拼进 system prompt），不在 prompt.txt 手动维护
- 返回结构用 `Result` / `ListResult` / `DictResult`，成败查 `.error is None`
- 保持轻量：新增文件必要时才建，能在现有模块内消化则内联

### 6. 与用户交替评估

用户是**评估者**，主动识别以下常见偏差：

- 引入了未经验证的想象（数字、术语、约束强度与原文有偏差）
- 完全照搬导致 docstring 或代码带走重架构假设
- 假深化：为已被现有机制解决的问题再加一层
- 触发反模式：即使外部项目这么做，kernel 场景下不适用

评估者的反馈闭环允许生成者从两个极端回到中间。**不追求一次到位**，接受"想象版 → 照搬版 → 提炼版"的演进路径。

## 常见陷阱

- **生成者忽略现有机制**：容易看到问题就加代码，不先问"现有的什么已经能解决它"。评估者应主动追问这一条。
- **数字与措辞偏差**：如把原文 "5-7 字" 加工成 "5-15 字"、把 "必须" 改成 "建议"。原文有明确用词时优先原文。
- **忽略架构差异**：如 Codex 的 tool call 天然记录参数，模型能从历史看到；kernel 的预置函数调用不留 tool_call 记录，需要显式回显。
- **示例膨胀**：为求"教育充分"堆叠多个示例。对照结构（好 vs 差）本身承担教学，数量堆叠是负担。

## 一个具体范例：plan 的三段演进

同一个函数三次迭代，展示"想象 → 照搬 → 提炼"路径。

### 版本 A（我的想象版，被否决）

```
每步 5-15 字
建议同时只有一个 in_progress
整表覆盖，无 add/delete
```

**问题**：数字（5-15）拍脑袋、约束强度（建议）弱化了原文，读者/模型看到的规范与成熟设计不一致。

### 版本 B（Codex 完全照搬版，被否决）

70 行 docstring，含使用时机 7 条 + 使用时避免 3 条 + 维护规范 6 条 + 高质量示例 3 个 + 低质量示例 3 个。

**问题**：Codex 是重型工具，规范密度对应它的训练目标；个人工具照搬会稀释关键信号。多数条目是同一原理的角度差异，未经合并。

### 版本 C（提炼版，采纳）

```
每步 5-7 字（沿用原文数字）
始终恰好一个 in_progress（沿用原文强制语气）
何时用：任务跨多轮、有逻辑阶段，或用户一句话要求做多件事
何时不用：单步任务；不填充；不列做不到的事
示例：1 个高质量 + 1 个低质量对照
```

**做法**：合并 Codex 的 22 条为 6 条核心 + 2 个对照示例。数字与措辞回归原文，删除废话，示例保留对照结构。规模压缩到 46 行。

## 与其他 skill 的关系

- 借鉴过程涉及具体代码变更时，动手前遵循 CLAUDE.md 的手术式修改原则
- 涉及反模式识别时，直接引用 CONTEXT.md §反模式 的五条命名
- 涉及测试新增时，遵循 kernel 现有 auto-discover（`globals()` 扫 `test_*`）约定

## 起点提示

用户明确请求"用 feature-borrow"时进入此流程。若用户没明确请求但请求形态匹配（如"研究 codex 的 X"），主动确认是否走此工作流。
